Parquet write issue on Windows (HADOOP_HOME / winutils)
=======================================================

Context
-------
- While running `src/processing/auth_to_parquet.py` to write the LANL auth dataset to Parquet (Silver layer), Spark failed with a `Py4JJavaError`.
- Root cause from the Java stack trace:
  - `java.io.FileNotFoundException: HADOOP_HOME and hadoop.home.dir are unset.`
  - `Did not find winutils.exe` warnings in the log.
- This happens when Spark (via Hadoop's RawLocalFileSystem) tries to create output / temp directories and set permissions on Windows.

Why this happens
----------------
- On Windows, Hadoop's filesystem code expects a native helper binary `winutils.exe` and a configured `HADOOP_HOME` directory.
- If `HADOOP_HOME` and `hadoop.home.dir` are not set, and `winutils.exe` is not found, directory creation with permission setting can fail with a hard error, not just a warning.
- Reads (CSV/bz2) often work with only WARN messages, but **writes** to filesystems (like Parquet output) go deeper into the Hadoop FS stack and can throw exceptions.

Effect on the project
---------------------
- All four LANL sources (auth, dns, flows, proc) are successfully ingested into Spark DataFrames and verified:
  - Auth: ~708M rows, 18 partitions.
  - DNS: ~40.8M rows, 1 partition.
  - Flows: schema as documented, sample rows verified.
  - Proc: ~426M process events, 1 partition.
- URLHaus acquisition works and writes `sample data/urlhaus_sample.json`.
- Attempting to write the Silver layer for auth to Parquet (e.g. `data/silver/auth/`) currently fails on this Windows machine due to the missing `HADOOP_HOME` / `winutils.exe` setup.

Options to resolve (future work)
--------------------------------
1. **Install Hadoop/winutils on Windows (recommended if continuing on this machine):**
   - Download a `winutils.exe` bundle that matches the Spark/Hadoop version.
   - Extract to a folder such as `C:\hadoop\bin\winutils.exe`.
   - Set environment variables:
     - `HADOOP_HOME=C:\hadoop`
     - Add `%HADOOP_HOME%\bin` to `PATH`.
   - Restart the terminal/IDE and rerun `auth_to_parquet.py` and similar scripts.

2. **Run Parquet writes on a Linux/macOS environment (e.g. WSL2, remote VM, cloud notebook):**
   - Clone the repository to a non-Windows environment.
   - Install Java, Spark, and Python dependencies.
   - Run the `*_to_parquet.py` scripts there; Hadoop's local filesystem usually works without extra configuration.

3. **Short-term workaround (for course milestones):**
   - For M2, it is sufficient to demonstrate ingestion and acquisition; Parquet (Silver layer) can be documented as planned/partially implemented.
   - For M3, if environment setup is still blocked, document this Windows-specific Hadoop limitation in the report and focus on DataFrame-level transformations (using the raw CSV/bz2 reads) while planning to move them to Parquet once the environment is ready.

Summary
-------
- The Parquet write failure is an environment configuration issue specific to Hadoop/Spark on Windows (missing `HADOOP_HOME` and `winutils.exe`), not a logic error in the auth ingestion or processing code.
- Ingestion of LANL auth/dns/flows/proc and URLHaus acquisition are confirmed working; enabling the Silver Parquet layer will require either configuring Hadoop on Windows or running the write jobs in a non-Windows environment.

Issues Faced:
1. JAR Version Mismatch ("Version Frankenstein")
PySpark ships with Hadoop 3.3 internals. Using hadoop-aws:3.4.0 introduced class files expecting methods only in Hadoop 3.4, while using hadoop-aws:3.3.6 still caused issues because Parquet's VectorIoBridge called VectoredReadUtils.validateNonOverlappingAndReturnSortedRanges() — a method added in Hadoop 3.4. This triggered NoSuchMethodError during every Parquet read from S3.
2. Vectored I/O Conflict
Modern Parquet readers use vectored I/O to read multiple file chunks simultaneously for speed. On Windows this failed both due to the version mismatch above and how the JVM interacts with native filesystem calls. Disabling it required targeting three layers simultaneously: JVM startup flags via spark.driver.extraJavaOptions and spark.executor.extraJavaOptions, Spark SQL config (spark.sql.parquet.enableVectorizedReader), and the Hadoop S3A filesystem property (spark.hadoop.fs.s3a.vectored.read.enabled). Earlier attempts using SPARK_SUBMIT_OPTS and runtime Spark configs alone failed because they don't reach the Parquet layer early enough on Windows.
3. Duration String Parsing Bug
Hadoop 3.3+ introduced human-readable duration strings like 60s and 24h in its default config files. The environment was calling getLong() instead of getDuration(), meaning the Java Long parser received strings with letters and threw NumberFormatException. Fixed by overriding all duration-style defaults with pure numeric millisecond values (e.g. 86400000 for 24 hours).
4. Windows Environment Initialization
PySpark on Windows fails to locate the correct Python executable and winutils.exe needed for local file operations out of the box. Additionally, HADOOP_HOME was set to C:\hadoop\bin instead of C:\hadoop, causing Spark to construct a doubled path \bin\bin\winutils.exe. Fixed by sanitizing the environment before SparkSession is built — forcing PYSPARK_PYTHON and PYSPARK_DRIVER_PYTHON to sys.executable, and normalizing HADOOP_HOME to strip the trailing \bin if present.
5. S3A Connector Not Included
PySpark doesn't ship with the S3A connector. The hadoop-aws and aws-java-sdk-bundle JARs had to be pulled in explicitly via spark.jars.packages, with Maven resolving and caching them via Ivy on first run.
6. Column Name and Path Mismatches
The silver Parquet schemas written to S3 used different names than the pipeline code assumed — source_computer instead of computer, source_user instead of user, S3 folder proc instead of processes. URLHaus stored full URLs rather than bare hostnames, so the join key required extracting the hostname with regexp_extract before joining against dst_computer. All of these required runtime inspection of actual schemas before the joins could be fixed.
Final Technical Baseline

Connector: hadoop-aws:3.3.6 matched to Spark's internal Hadoop version
I/O mode: legacy sequential reader with vectored I/O disabled at all three layers
Config style: pure numeric values, no time-unit suffixes
Windows bootstrap: explicit Python executable and normalized HADOOP_HOME set before JVM initialization
